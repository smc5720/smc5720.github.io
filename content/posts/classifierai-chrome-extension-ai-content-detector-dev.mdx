## 소개

AI는 인터넷의 모든 것을 변화시켰습니다. 이전보다 빠른 정보 접근부터 개발 워크플로에 AI를 통합하는 것까지 그 영향력은 광범위합니다.

AI 사용의 긍정적인 측면과 함께 부정적인 측면도 존재합니다. 현재 인터넷에는 사이트의 Google 검색 관련성을 높이기 위해 순수하게 AI가 생성한 콘텐츠가 대량으로 생산되고 있으며, 이는 불필요한 경쟁을 유발합니다. 또한 AI의 발전으로 인해 AI가 생성한 콘텐츠와 인간이 작성한 콘텐츠를 구별하는 것이 점점 더 어려워지고 있습니다. AI 자체는 사람들이 이를 사용하는 방식 때문에 나쁜 이미지를 갖게 되었습니다.

AI가 어디에나 존재하는 상황에서도 많은 사람들이 이에 대응하고 있습니다. DEV 사용자로서의 목표는 dev.to의 AI 생성 기사를 최소화하여 사용자 경험을 향상시키고 개발자로서 배우고 성장할 수 있는 다양한 콘텐츠를 제공하는 것입니다.

이러한 배경 하에 Elmar와의 협업으로 **ClassifierAI**를 개발했습니다.

![ClassifierAI 로고](https://dev-to-uploads.s3.amazonaws.com/uploads/articles/ovhtick339ll39h5bx8i.png)

> **고지사항:** 이 프로젝트는 프로토타입입니다. dev.to 외부의 데이터를 사용하기 때문에 모델이 정확하지 않습니다. 이 크롬 확장은 AI 생성 콘텐츠를 완벽하게 필터링하는 솔루션으로 사용되도록 의도된 것이 아닙니다.

## 무엇을 만들었는가

**ClassifierAI**는 DEV의 AI 생성 콘텐츠를 즉시 감지할 수 있는 **프로토타입** 크롬 확장입니다. **TensorFlow.js**를 사용하여 다음 두 가지 기능을 수행합니다:

- **이미지 분류**: Google의 [Teachable Machine](https://teachablemachine.withgoogle.com/)을 사용하여 기사의 커버 이미지가 AI 생성인지 판별합니다.
  - 학습에 사용된 데이터셋: [HuggingFace AI-Generated-vs-Real-Images-Datasets](https://huggingface.co/datasets/Hemg/AI-Generated-vs-Real-Images-Datasets)
  - 총 866개 이미지(클래스당 433개, "AI"와 "NotAI" 두 클래스)로 학습

```plaintext
866 Images Total
2 Classes Total
  - "AI" Class
  - "NotAI" Class

Epochs: 30
Batch Size: 16
Learning Rate: 0.0001
```

- **텍스트 분류**: 기사 자체가 AI 생성인지 판별합니다.
  - 사용된 데이터셋: [wikipedia-human-ai](https://huggingface.co/datasets/gouwsxander/wikipedia-human-ai)

초기에는 Google 이미지에서 100개의 이미지를 순차적으로 스캔하여 AI 생성 여부를 레이블링하는 크롬 확장으로 시작했습니다. 이번 버전에서는 dev.to 전용으로 업데이트되어 커버 이미지를 포함한 전체 기사를 분류하고 결과를 사용자에게 보여줍니다. 이 크롬 확장은 사용자가 DEV의 기사를 클릭할 때마다 동작합니다.

## 이 프로젝트가 존재하는 이유

생성형 AI는 현재 뜨거운 주제입니다. 많은 동료들이 생성형 AI를 싫어하는 이유는 "일자리를 대체할 것"이라는 인식 때문입니다.

![일자리 대체 우려](https://dev-to-uploads.s3.amazonaws.com/uploads/articles/7jgrx6qe4qyvguw2rf1o.png)

**GPTZero**와 같은 도구들이 있지만, 여전히 문제가 존재합니다:

- AI 생성 여부를 확인하려면 복사/붙여넣기를 해야 하는 **많은 수작업**이 필요합니다.
- GPTZero를 우회하여 "Human" 결과를 얻는 도구들이 있습니다. 커뮤니티마다 작성 스타일이 다르기 때문에 시간이 지남에 따라 AI 생성 콘텐츠를 식별하기가 더 어려워집니다.

이러한 문제들을 고려하여, **GPTZero를 열고 복사/붙여넣기 없이 AI 생성 콘텐츠를 자동으로 식별하는 도구**를 만들고자 했습니다.

또한, 이 크롬 확장은 DEV 전용으로 맞춤화되어 있으며, 향후 수집될 데이터셋이 더욱 집중적이고 정확할 것이라고 믿습니다.

시작하려면 컴퓨터에 빈 폴더를 만들고 VSCode 또는 다른 IDE로 열어주세요.

**1) 터미널에서 저장소를 클론합니다:**

```
git clone https://github.com/FrancisTR/ClassifierAI.git
```

```
cd ClassifierAI
```

**3) 패키지를 설치하고 빌드를 실행합니다:**

```
npm install
npm run build
```

**4) 브라우저에서 확장 페이지로 이동하여 개발자 모드를 활성화합니다.**

초기에는 커버 이미지 스캔에 Teachable Machine으로 학습한 커스텀 모델 대신 Gemma 4를 사용하려 했습니다. 그러나 로컬 모델의 성능 문제로 크롬 확장이 브라우저 전체를 먹통으로 만드는 사태가 발생했습니다. 결국 기존 LLM 대신 이미지 분류를 위한 기존 데이터셋을 사용하여 모델을 학습시키는 방향으로 전환했습니다.

사용된 이미지 데이터셋은 HuggingFace의 [AI-Generated-vs-Real-Images-Datasets](https://huggingface.co/datasets/Hemg/AI-Generated-vs-Real-Images-Datasets)입니다. 데이터셋의 이미지가 DEV에서 사용되는 이미지와 매우 다르기 때문에 모델이 완벽하지 않지만, 이전 데이터셋보다는 다양합니다.

새로 구현한 기능은 기사를 분류하여 AI 생성 여부를 판별하는 것입니다. 텍스트 분류는 기사의 전체 텍스트를 추출하여 모델에 입력합니다:

```javascript
function getCleanArticleText() {
  const root =
    document.querySelector(".crayons-article__body") ||
    document.querySelector("article");

  return root ? normalizeWhitespace(root.innerText) : "";
}
```

그런 다음 Wikipedia 데이터셋을 기반으로 분류를 수행하고 인간 작성 대 AI 생성 비율을 사용자에게 제공합니다:

```javascript
function detectGPTStyle(text) {
  if (!wikipediaDataset || wikipediaDataset.length < 10) {
    return baseUnknownResult();
  }

  const datasetScore = compareDataset(text);       // AI %
  const devHuman = computeDevHumanScore(text);     // Human %
  const generalHuman = computeGeneralScore(text);  // Human %
  const aiPenalty = detectAIPatterns(text);        // AI %

  let humanScore =
    devHuman * 0.40 +
    (100 - datasetScore) * 0.35 +
    generalHuman * 0.25 -
    aiPenalty * 0.25;

  humanScore = squashScore(humanScore);

  let label =
    humanScore <= 33.33
      ? "AI-generated"
      : humanScore >= 66.66
        ? "Human-written"
        : "Mixed";

  const finalScore = Number(humanScore.toFixed(2));
  return {
    label,
    averageAIScore: finalScore,
    humanPercent: finalScore,
    aiPercent: Number((100 - humanScore).toFixed(2)),
    mixedPercent:
      finalScore > 33.33 && finalScore < 66.66 ? 100 : 0,
  };
}
```

UI 개선으로 **chart.js**를 사용하여 정보를 더 깔끔하게 표시합니다.

## GitHub Copilot 사용 경험

> **고지사항:** 이 섹션은 GitHub Copilot 사용 경험에 관한 내용입니다. Elmar는 개발 워크플로에 ChatGPT를 사용하며, 코드베이스 탐색과 버그 수정에 멘토로 활용했습니다.

GitHub Copilot은 주로 **마이그레이션 과정**에서 유용하게 활용되었습니다:

- **ml5.js에서 TensorFlow.js로** 이전
- Chrome Extension Manifest V3 준수 확인

예를 들어, GitHub Copilot에게 "ml5.js에서 tensorflow.js로 마이그레이션"을 요청하자 작업을 완료해 주었습니다.

> 다른 사람의 코드베이스에서 작업하는 것은 완전히 다른 도전이었습니다.
>
> 때로는 마치 유리 위를 걷는 것처럼 무언가를 망칠까봐 두려웠습니다.
>
> 코드베이스를 먼저 읽어야 했고, ChatGPT가 개념과 낯선 코드 부분을 이해하는 데 도움을 주었습니다.
>
> Frontend Mentor를 통해 다른 사람의 코드를 리뷰한 경험이 있었지만, 실제 프로젝트에 기여하는 것은 달랐습니다. 모든 변경사항에는 결과가 따랐습니다.
>
> 코드베이스 자체보다 더 많이 배운 것은 소통의 중요성이었습니다. 아이디어를 논의하고, 질문하고, 메인테이너에게 확인을 구하는 데 많은 시간을 보냈습니다.
>
> 적응력의 중요성도 배웠습니다. 프로젝트가 기여 과정에서 발전하면서 가정과 계획도 함께 발전해야 했습니다. 또한 첫 번째 실제 병합 충돌을 경험했지만 결국 해결했습니다.
>
> 전반적으로 ClassifierAI에 기여하면서 실제 소프트웨어 개발이 혼자 프로젝트를 만드는 것과 얼마나 다른지 훨씬 더 잘 이해하게 되었습니다.
>
> 이 경험은 내가 커리어 초반에 읽었던 것을 다시 한 번 증명했습니다:
>
> *"코드를 읽고 이해하는 능력은 종종 코드를 작성하는 능력보다 더 가치 있다."*

### 크롬 확장의 현재 과제

이 프로젝트가 프로토타입인 이유는 여러 가지입니다:

- **모델과 데이터셋이 dev.to에 맞지 않습니다.** 텍스트 데이터셋은 Wikipedia에서, 이미지 데이터셋은 HuggingFace에서 가져왔으며 둘 다 dev.to 플랫폼에 정확하지 않습니다. 향후 더 많은 협업을 통해 dev.to의 데이터를 수집하고 모델을 재학습할 계획입니다.

- **번역이나 문법 검사 등 AI를 적절하게 사용하는 경우를 고려하지 않으며**, 기사의 어느 부분이 AI 생성인지 보여주는 기능이 제한적입니다.
